1. C'est quoi, un sous-agent ?

Analogie simple
Imaginez un chef cuisinier (Claude) qui prépare un repas complexe. Plutôt que de faire tout seul — couper les légumes, faire la sauce, cuire la viande — il peut appeler des commis de cuisine. Chaque commis reçoit une tâche précise, travaille de son côté, puis rapporte le résultat au chef. Le chef assemble le tout.

Un sous-agent est une instance séparée de Claude, lancée par Claude lui-même pour accomplir une tâche spécifique. Le sous-agent :

  • Reçoit un briefing (ce qu'il doit faire, pourquoi, dans quel contexte)
  • A accès aux mêmes outils (lire des fichiers, chercher du code, etc.)
  • Travaille de façon indépendante — il ne voit pas la conversation principale
  • Renvoie un résultat unique au Claude principal quand il a fini

Le mot clé est parallèle : plusieurs sous-agents peuvent travailler en même temps, chacun sur une tâche différente. C'est comme avoir plusieurs onglets ouverts, mais chacun avec son propre assistant.

Sous-agent vs. outil

CaractéristiqueOutil (ex. lire un fichier)Sous-agent
Fait une seule action✅ Oui❌ Non — peut enchaîner plusieurs actions
Prend des décisions❌ Non✅ Oui — il réfléchit et s'adapte
Voit la conversation✅ Oui (indirectement)❌ Non — repart de zéro avec son briefing
Peut tourner longtemps❌ Non (rapide)✅ Oui — peut faire des dizaines d'étapes

2. Pourquoi des sous-agents parallèles ?

❌ Sans sous-agents

Claude fait tout séquentiellement :

  1. Chercher les fichiers TypeScript → attendre
  2. Chercher les fichiers Python → attendre
  3. Analyser l'architecture → attendre
  4. Vérifier les tests → attendre

Durée totale : la somme de tout.

✅ Avec sous-agents parallèles

Claude lance 4 sous-agents en même temps :

  1. Agent A → fichiers TypeScript
  2. Agent B → fichiers Python
  3. Agent C → architecture
  4. Agent D → tests

Durée totale : celle du plus lent des 4.

Gain concret
Si chaque recherche prend 30 secondes, le mode séquentiel met 2 minutes. Avec 4 sous-agents parallèles, on tombe à ~30 secondes. C'est 4× plus rapide.

Quand c'est utile

  • Explorer un gros projet — chercher dans plusieurs dossiers en même temps
  • Analyser plusieurs aspects — sécurité + performance + qualité en parallèle
  • Comparer des approches — tester 3 stratégies différentes simultanément
  • Recueillir des informations — lire la doc, le code et les tests en même temps

3. Les types de sous-agents

Claude Code propose plusieurs types de sous-agents spécialisés :

TypeSpécialitéQuand l'utiliser
Explore Explorer le code source rapidement Trouver des fichiers, chercher des mots-clés, comprendre la structure d'un projet
Plan Concevoir des stratégies d'implémentation Réfléchir à l'architecture, identifier les fichiers critiques, peser le pour et le contre
general-purpose Tout faire (recherche, code, exécution) Tâches complexes qui nécessitent plusieurs étapes

Choisir le bon type

Règle simple
  • Vous cherchez quelque chose → Explore
  • Vous planifiez quelque chose → Plan
  • Vous devez faire quelque chose de complexe → general-purpose

4. Comment les déclencher

En tant qu'utilisateur, vous n'avez pas besoin de syntaxe spéciale. Claude décide automatiquement quand utiliser des sous-agents. Mais vous pouvez l'encourager avec votre formulation.

Formulations qui déclenchent naturellement des sous-agents

✅ Déclenche des sous-agents parallèles

Analyse ce projet : vérifie la sécurité, la performance
et la qualité du code en parallèle.
Cherche tous les endroits où on utilise l'authentification,
dans le backend ET le frontend en même temps.
Compare trois approches pour implémenter cette fonctionnalité
et dis-moi laquelle est la meilleure.

❌ Ne déclenche probablement PAS de sous-agents

Lis le fichier config.ts

→ Trop simple, un seul outil suffit.

Corrige ce bug dans la ligne 42

→ Tâche ciblée, pas besoin de parallélisme.

Le mot magique : « en parallèle »

Si vous écrivez explicitement « en parallèle » ou « en même temps », Claude est fortement incité à lancer plusieurs sous-agents simultanément.

Exemple

Lance en parallèle :
1. Un agent qui explore la structure du backend
2. Un agent qui liste toutes les routes API
3. Un agent qui vérifie la couverture de tests

5. Exemples concrets

Exemple 1 — Explorer un projet inconnu

Situation : Vous venez de rejoindre un projet et vous voulez comprendre comment il fonctionne.

Ce que vous dites à Claude :

Je découvre ce projet. Peux-tu m'expliquer en parallèle :
- La structure des dossiers et l'architecture générale
- Comment l'authentification fonctionne
- Quels sont les principaux modèles de données
- Comment les tests sont organisés

Ce que Claude fait :

  1. Lance 4 sous-agents Explore en parallèle
  2. Chacun fouille le code de son côté
  3. Claude reçoit les 4 rapports
  4. Il vous fait une synthèse claire et structurée

Exemple 2 — Audit de sécurité rapide

Situation : Vous voulez vérifier la sécurité avant de mettre en production.

Ce que vous dites à Claude :

Fais un audit de sécurité en parallèle :
- Vérifie les injections SQL possibles
- Cherche les mots de passe en dur dans le code
- Analyse la gestion des permissions
- Vérifie que les entrées utilisateur sont validées

Ce que Claude fait :

  1. Lance 4 sous-agents en parallèle, chacun spécialisé sur un aspect
  2. Chaque agent cherche dans tout le projet
  3. Claude compile les résultats en un rapport avec les risques classés par gravité

Exemple 3 — Refactoring à grande échelle

Situation : Vous voulez renommer un concept dans tout le projet.

Ce que vous dites à Claude :

Je veux renommer "User" en "Account" partout dans le projet.
D'abord, recherche en parallèle tous les endroits
où "User" apparaît : modèles, routes, tests, documentation.

Ce que Claude fait :

  1. Lance des sous-agents pour chercher dans chaque zone du projet
  2. Compile la liste complète des fichiers à modifier
  3. Vous propose un plan de renommage avant de toucher au code

6. Sous le capot

Voici comment les sous-agents fonctionnent techniquement (pas besoin de retenir tout, c'est pour comprendre).

Le cycle de vie d'un sous-agent

Points techniques importants

AspectDétail
Mémoire Un sous-agent ne voit pas la conversation principale. Il repart de zéro avec uniquement son briefing. C'est pourquoi le briefing doit être complet.
Outils Les sous-agents ont accès aux mêmes outils que Claude principal (lire des fichiers, chercher, etc.) mais un agent Explore ne peut pas modifier de fichiers.
Isolation Un sous-agent peut travailler dans un worktree isolé — une copie séparée du dépôt Git, pour éviter les conflits entre agents qui modifient des fichiers.
Résultat Chaque sous-agent renvoie un seul message de résultat. Claude principal ne voit pas les étapes intermédiaires.
Foreground vs Background Foreground (par défaut) : Claude attend le résultat avant de continuer. Background : Claude continue à travailler pendant que l'agent tourne.

Le briefing — la clé de tout

La qualité du résultat dépend directement de la qualité du briefing. Un bon briefing :

❌ Mauvais briefing

"Cherche les bugs"

Trop vague — le sous-agent ne sait pas où chercher ni quoi considérer comme un bug.

✅ Bon briefing

"Cherche dans src/api/ tous les
endpoints qui ne valident pas les
entrées utilisateur. Vérifie
particulièrement les paramètres
de requête et les corps JSON.
Rapporte chaque fichier et ligne
concernés."

Un bon briefing contient :

  1. Le contexte — pourquoi cette tâche est nécessaire
  2. L'objectif précis — ce qu'on cherche exactement
  3. Le périmètre — où chercher
  4. Le format attendu — comment présenter le résultat

7. Bonnes pratiques

✅ À faire

  • Séparez les tâches indépendantes — si deux recherches n'ont aucun lien, elles peuvent tourner en parallèle
  • Soyez précis dans vos demandes — « en parallèle, vérifie X, Y et Z » est plus clair que « vérifie tout »
  • Utilisez les sous-agents pour la recherche — c'est leur point fort, surtout dans les gros projets
  • Laissez Claude synthétiser — ne demandez pas aux sous-agents de prendre des décisions finales, laissez le Claude principal assembler

❌ À éviter

  • Ne lancez pas de sous-agent pour une tâche simple — lire un fichier ne nécessite pas un agent dédié
  • Ne créez pas de dépendances entre sous-agents — si l'agent B a besoin du résultat de l'agent A, ils ne peuvent pas tourner en parallèle
  • Ne demandez pas à un sous-agent de modifier ce qu'un autre sous-agent modifie — les conflits de fichiers sont possibles
  • N'attendez pas qu'un sous-agent « se souvienne » — chaque sous-agent repart de zéro

Tableau de décision

SituationSous-agents ?Pourquoi
Lire un fichier précis❌ NonUn simple outil suffit
Chercher un mot dans le code❌ NonGrep est plus rapide
Explorer 4 aspects d'un projet✅ Oui4 recherches indépendantes
Comparer 3 stratégies✅ Oui3 analyses indépendantes
Modifier un fichier puis le tester❌ NonSéquentiel, pas parallèle
Auditer sécurité + perf + qualité✅ Oui3 domaines indépendants

8. Limites et pièges

⚠️ Points d'attention

Les sous-agents ne communiquent pas entre eux

Chaque sous-agent est isolé. Si l'agent A découvre quelque chose d'important pour l'agent B, il ne peut pas le lui dire. C'est Claude principal qui fait le lien après coup.

Le briefing ne contient pas la conversation

Un sous-agent ne voit rien de ce que vous avez dit avant. Si vous avez donné du contexte important au début de la conversation, Claude doit le ré-inclure dans le briefing du sous-agent.

Conflit de fichiers

Si deux sous-agents essaient de modifier le même fichier en même temps, les résultats peuvent se chevaucher. Solution : utiliser l'isolation worktree (chaque agent travaille sur sa propre copie du code).

Coût en ressources

Chaque sous-agent consomme des tokens (la « monnaie » de l'IA). Trois sous-agents en parallèle consomment ~3× plus qu'un seul Claude. Mais c'est souvent rentable : 3× plus de tokens pour 3× moins de temps d'attente.

Résultats partiels

Si un sous-agent échoue ou ne trouve rien, Claude principal le signale et peut relancer la recherche avec un briefing ajusté. Votre travail n'est pas perdu.

9. Récapitulatif

ConceptEn une phrase
Sous-agentUne instance séparée de Claude qui reçoit une tâche et renvoie un résultat
ParallèlePlusieurs sous-agents qui travaillent en même temps sur des tâches indépendantes
BriefingLe message qui explique au sous-agent ce qu'il doit faire — doit être autonome et complet
ExploreType de sous-agent spécialisé dans la recherche de code
PlanType de sous-agent spécialisé dans la conception et l'architecture
ForegroundClaude attend le résultat avant de continuer
BackgroundClaude continue à travailler pendant que l'agent tourne en arrière-plan
WorktreeCopie isolée du code pour éviter les conflits entre agents
En résumé
Les sous-agents parallèles sont comme une équipe : chaque membre travaille sur sa spécialité, et le chef d'équipe (Claude) assemble le tout. Vous n'avez pas besoin de gérer l'équipe — il suffit de décrire ce que vous voulez, et Claude décide comment répartir le travail.
