Plans et Boucles dans Claude Code

Faire réfléchir Claude avant d'agir, puis le faire tourner en continu.

Claude Code peut faire bien plus qu'exécuter une instruction à la fois. Deux fonctions puissantes vous permettent de passer à la vitesse supérieure : le mode Plan (demander à Claude de réfléchir avant de coder) et les Boucles (faire tourner une tâche en continu, automatiquement). Ce guide vous explique tout, du concept à la pratique.

Partie 1 — Le Mode Plan

C'est quoi, le mode Plan ?

Imaginez que vous demandez à un artisan de rénover votre cuisine. Deux approches sont possibles :

  • Sans plan : il commence à casser les murs tout de suite. C'est rapide au début, mais il risque de se rendre compte à mi-chemin qu'il a oublié la plomberie.
  • Avec plan : il fait d'abord un croquis, liste le matériel, identifie les étapes, et ensuite il commence les travaux.

Le mode Plan de Claude Code, c'est exactement ça. Quand vous l'activez, Claude va :

  1. Explorer votre code pour comprendre ce qui existe
  2. Analyser ce que vous demandez
  3. Proposer un plan détaillé (quels fichiers modifier, dans quel ordre, quelles précautions prendre)
  4. Attendre votre accord avant de toucher au code
En une phrase : le mode Plan sépare la réflexion de l'action. Claude pense d'abord, code ensuite.

Pourquoi planifier avant de coder ?

Sans planification, Claude a tendance à foncer tête baissée. Sur des tâches simples (corriger une faute de frappe, ajouter un bouton), ce n'est pas un problème. Mais dès que la tâche est plus complexe, planifier évite :

Problème sans planSolution avec plan
Claude modifie le mauvais fichierLe plan liste précisément les fichiers à toucher
Il casse une fonctionnalité existanteLe plan identifie les dépendances et les risques
Il oublie une étape importanteLe plan décompose la tâche en étapes séquentielles
Vous ne comprenez pas ce qu'il faitLe plan vous donne une vision claire avant qu'il commence
Le résultat ne correspond pas à ce que vous vouliezVous pouvez corriger le plan avant qu'il ne code
Règle d'or : Plus la tâche est complexe, plus le plan est utile. Pour une modification d'une ligne, pas besoin de plan. Pour restructurer un projet entier, c'est indispensable.

Comment utiliser le mode Plan

Méthode 1 : la commande /plan

Dans Claude Code, tapez simplement :

/plan

Claude entre en mode planification. Il ne va rien modifier — il va seulement lire, analyser et proposer. Vous pouvez aussi combiner avec votre demande :

Peux-tu planifier comment ajouter un système de notifications à mon application ?

Claude comprendra qu'il doit d'abord réfléchir.

Méthode 2 : demander explicitement

Vous pouvez aussi simplement demander en langage naturel :

Avant de coder, fais-moi un plan détaillé pour cette fonctionnalité.
Ne touche à rien pour l'instant, dis-moi juste ce que tu ferais.

Étapes typiques d'une session avec plan

  1. Vous décrivez ce que vous voulez
  2. Claude explore le code existant (il lit les fichiers, cherche les patterns)
  3. Claude vous présente un plan structuré
  4. Vous relisez le plan, posez des questions, demandez des ajustements
  5. Quand le plan vous convient, vous dites OK, exécute le plan
  6. Claude code en suivant le plan étape par étape

Anatomie d'un plan

Un bon plan de Claude contient généralement :

Exemple de plan type : 
## Plan : Ajouter un mode sombre à l'application

### Contexte
L'application utilise actuellement un thème clair codé en dur
dans style.css. Les couleurs sont définies dans des variables
CSS dans :root.

### Étapes

1. **Définir les variables du mode sombre**
   - Fichier : style.css
   - Ajouter un bloc `[data-theme="dark"]` avec les couleurs inversées
   - Garder les mêmes noms de variables

2. **Ajouter un bouton de basculement**
   - Fichier : index.html
   - Placer un bouton dans la barre de navigation
   - Icône soleil/lune

3. **Écrire le JavaScript de basculement**
   - Fichier : script.js (nouveau fichier)
   - Lire la préférence sauvegardée dans localStorage
   - Basculer l'attribut data-theme sur <html>

4. **Respecter la préférence système**
   - Détecter prefers-color-scheme au chargement

### Risques identifiés
- Les SVG inline utilisent des couleurs en dur → les adapter
- Le tableau en étape 4 a un fond blanc fixe → le rendre dynamique

### Fichiers impactés
- style.css (modifié)
- index.html (modifié)
- script.js (créé)

Remarquez que le plan identifie :

  • Le contexte — ce qui existe déjà
  • Les étapes — dans l'ordre logique
  • Les fichiers — lesquels seront touchés et comment
  • Les risques — ce qui pourrait mal tourner

Bonnes pratiques

✓ À faire

  • Donnez du contexte — Plus Claude comprend votre projet, meilleur sera le plan. Ex : « C'est une app mobile pour des couples, on utilise Kotlin et Jetpack Compose »
  • Relisez le plan attentivement — C'est le moment de corriger le tir, avant que du code soit écrit
  • Posez des questions — Si une étape n'est pas claire, demandez des précisions
  • Demandez des alternatives — « Y a-t-il une autre approche ? »
  • Validez par morceaux — Sur un gros plan, demandez à Claude d'exécuter une étape à la fois

✗ À éviter

  • Planifier des tâches triviales — Corriger une faute de frappe ne nécessite pas de plan
  • Ignorer le plan — Si vous dites juste « OK fais-le » sans lire, vous perdez l'intérêt du mode Plan
  • Changer d'avis en cours de route — Si vous voulez modifier le plan, mieux vaut le re-planifier plutôt que corriger à la volée

Sous le capot : comment ça marche

Quand vous activez le mode Plan, voici ce qui se passe en coulisses :

  1. Restriction des outils — Claude Code désactive temporairement les outils d'écriture (édition de fichiers, exécution de commandes). Claude ne peut que lire et réfléchir.
  2. Exploration du code — Claude utilise ses outils de recherche pour parcourir votre projet : il lit les fichiers, cherche des patterns, comprend la structure.
  3. Raisonnement structuré — Claude utilise son espace de réflexion (son « thinking » interne, invisible pour vous) pour organiser ses idées. Il évalue différentes approches et choisit la meilleure.
  4. Présentation du plan — Claude vous montre le résultat de sa réflexion sous forme de plan lisible.
  5. Exécution contrôlée — Quand vous validez, les outils d'écriture sont réactivés et Claude suit le plan point par point.
Le saviez-vous ? Le plan lui-même est stocké dans la conversation. Claude peut s'y référer à tout moment pendant l'exécution. Si vous lui dites « on en est où dans le plan ? », il saura vous répondre.

Plans et agents spécialisés

Claude Code peut aussi déléguer la planification à un agent spécialisé (un « sous-agent Plan »). Cet agent :

  • A accès aux mêmes outils de lecture que Claude
  • Travaille dans un contexte séparé (il ne « pollue » pas la conversation principale)
  • Renvoie un plan structuré à Claude, qui vous le présente

C'est utile pour les très gros projets où l'exploration prend beaucoup de place dans la conversation.

Exemples concrets

Exemple 1 : Ajouter une fonctionnalité

Vous :

Je veux ajouter un système de favoris à mon application.
Les utilisateurs doivent pouvoir mettre en favori des articles
et les retrouver dans un onglet dédié.
Planifie-moi ça avant de coder.

Claude va : explorer votre base de données, vos modèles, vos écrans existants, puis proposer un plan couvrant le modèle de données, l'API, l'interface utilisateur et les tests.

Exemple 2 : Refactoring

Vous :

/plan
Mon fichier utils.js fait 800 lignes. Propose-moi un
découpage en modules plus petits sans rien casser.

Claude va : analyser les imports/exports, identifier les groupes de fonctions liées, vérifier qui appelle quoi, et proposer un découpage avec le détail des déplacements de fonctions.

Exemple 3 : Corriger un bug complexe

Vous :

L'app crash quand un utilisateur n'a pas de photo de profil.
Avant de toucher au code, analyse le problème et dis-moi
d'où ça vient et comment tu corrigerais.

Claude va : chercher les endroits où la photo de profil est utilisée, identifier les accès non protégés (null checks manquants), et proposer un plan de correction fichier par fichier.

Partie 2 — Les Boucles (/loop)

C'est quoi, une boucle ?

Une boucle, c'est demander à Claude de répéter une tâche automatiquement, à intervalle régulier ou en se régulant lui-même.

Pensez à un gardien de nuit qui fait sa ronde toutes les heures : il vérifie les portes, note les problèmes, et recommence. La commande /loop transforme Claude en ce gardien pour votre code.

En une phrase : /loop fait tourner Claude en continu sur une tâche, sans que vous ayez à relancer la demande à chaque fois.

Pourquoi utiliser des boucles ?

Les boucles sont utiles quand :

  • Vous attendez un résultat — Un build qui compile, des tests qui passent, un déploiement qui finit
  • Vous voulez surveiller quelque chose — Vérifier qu'un serveur reste en ligne, qu'une API répond
  • Vous avez une tâche répétitive — Vérifier la qualité du code toutes les 10 minutes pendant que vous développez
  • Vous voulez de l'autonomie — Laisser Claude travailler seul sur une liste de tâches pendant que vous faites autre chose

Comment lancer une boucle

Syntaxe de base

/loop 5m Vérifie si le build est passé et dis-moi le résultat

Cela demande à Claude de vérifier le build toutes les 5 minutes.

Format de l'intervalle

FormatSignificationExemple
30s30 secondes/loop 30s vérifie le serveur
5m5 minutes/loop 5m relance les tests
1h1 heure/loop 1h fais un rapport
(rien)Claude décide/loop surveille le déploiement
Limites : L'intervalle minimum est de 60 secondes et le maximum est de 60 minutes (3600 secondes). Si vous demandez moins ou plus, Claude ajustera automatiquement.

Arrêter une boucle

Pour arrêter une boucle en cours, vous pouvez :

  • Envoyer un nouveau message à Claude (la boucle s'interrompt)
  • Fermer la session Claude Code
  • Appuyer sur Ctrl+C

Les deux types de boucles

1. Boucle à intervalle fixe

Vous spécifiez l'intervalle :

/loop 5m Vérifie si les tests passent

Claude se réveille exactement toutes les 5 minutes et exécute la tâche. Simple et prévisible.

2. Boucle dynamique (auto-régulée)

Vous omettez l'intervalle :

/loop Surveille le déploiement et dis-moi quand c'est fini

Claude décide lui-même à quelle fréquence vérifier. C'est la méthode la plus intelligente : Claude adapte son rythme en fonction de ce qu'il observe.

Exemple concret : Si Claude surveille un build qui vient de démarrer, il va peut-être attendre 4 minutes avant le premier check (le temps que le build avance), puis 2 minutes entre les checks suivants s'il voit que ça progresse, puis s'arrêter quand c'est terminé.

Cas d'usage courants

Surveiller un build ou un déploiement

/loop 3m Vérifie le statut du build GitHub Actions et préviens-moi quand c'est fini

Parfait quand vous avez lancé un build et voulez être prévenu sans rester devant l'écran.

Vérification continue de la qualité

/loop 10m Lance les tests unitaires et signale tout échec

Utile pendant une session de développement intense où vous voulez un filet de sécurité.

Surveillance d'un processus

/loop Surveille l'import des données. Quand les 50 000 lignes sont traitées, génère un rapport.

Claude adapte son rythme et s'arrête quand c'est fini.

Tâche autonome avec progression

/loop Migre les fichiers du dossier legacy/ un par un vers le nouveau format. Fais un commit après chaque fichier.

Claude travaille seul, à son rythme, en faisant des pauses entre chaque étape.

Combiner avec une commande

/loop 5m /commit

Crée un commit automatique toutes les 5 minutes avec les changements en cours. Pratique pour sauvegarder régulièrement pendant une longue session.

Sous le capot : comment Claude gère le rythme

Comprendre comment les boucles fonctionnent en interne vous aidera à mieux les utiliser.

Le mécanisme de réveil

  1. Vous lancez /loop
  2. Claude exécute la tâche une première fois
  3. À la fin, Claude programme un réveil (comme une alarme) pour la prochaine exécution
  4. Claude « s'endort » — il ne consomme pas de ressources entre les réveils
  5. Au réveil, il reprend la tâche avec tout le contexte de la conversation

Le cache de contexte (important pour le coût)

Claude utilise un cache de conversation qui dure 5 minutes. Voici pourquoi c'est important :

  • Si le réveil est dans moins de 5 minutes — le cache est encore « chaud ». Claude reprend instantanément et le coût est faible.
  • Si le réveil est dans plus de 5 minutes — le cache a expiré. Claude doit relire toute la conversation, ce qui est plus lent et plus coûteux.
Les bons intervalles selon le cache :
IntervalleCacheIdéal pour
1–4 minutes✓ ChaudSurveillance active (build en cours, tests)
5 minutes pile✗ Le pire choixÉvitez ! Vous payez le cache froid sans bénéfice
10–30 minutes✗ FroidTâches lentes où checker souvent ne sert à rien

Comment Claude choisit l'intervalle en mode dynamique

Quand vous lancez /loop sans intervalle, Claude prend en compte :

  • La nature de la tâche — un build de 2 minutes vs un déploiement de 20 minutes
  • Ce qu'il a observé — si rien n'a changé, il espace les vérifications
  • Le coût — il évite de gaspiller des ressources en vérifiant trop souvent
  • L'urgence — une surveillance critique sera plus fréquente

En pratique, pour une tâche sans urgence particulière, Claude choisit souvent un intervalle de 20 à 30 minutes.

Bonnes pratiques et pièges

✓ À faire

  • Soyez précis dans la tâche — « Vérifie si les tests passent » est mieux que « Surveille le projet »
  • Définissez une condition d'arrêt — « …et arrête-toi quand le build est vert »
  • Utilisez le mode dynamique quand vous ne savez pas combien de temps ça va prendre
  • Privilégiez des intervalles de 2–4 minutes pour la surveillance active (cache chaud)
  • Utilisez 15–30 minutes pour les tâches non urgentes (un seul cache froid rentabilisé)

✗ À éviter

PiègePourquoi c'est un problèmeAlternative
Intervalle de 5 minutesPire choix possible (cache expiré sans bénéfice)4 min (cache chaud) ou 15 min (cache froid rentabilisé)
Boucle sans condition d'arrêtTourne indéfiniment, consomme des ressourcesAjoutez « arrête quand… »
Intervalle trop court pour une tâche longue8 checks d'un build de 8 min = gaspillageUtilisez le mode dynamique
Tâche trop vagueClaude ne sait pas quoi vérifierSoyez spécifique sur ce qu'il doit regarder

Partie 3 — Combiner Plans et Boucles

Les plans et les boucles sont encore plus puissants ensemble. Voici des scénarios typiques :

Scénario 1 : Planifier puis exécuter en boucle

Étape 1 — Planifier :

Planifie la migration de tous les fichiers JavaScript en TypeScript.
Liste chaque fichier à migrer et l'ordre de migration.

Étape 2 — Valider le plan (vous relisez, ajustez)

Étape 3 — Exécuter en boucle :

/loop Migre le prochain fichier du plan. Fais un commit après
chaque fichier. Arrête-toi quand tous les fichiers sont migrés.

Claude suit le plan et migre un fichier par itération, avec des pauses entre chaque pour ne pas tout faire d'un bloc.

Scénario 2 : Plan + surveillance

Étape 1 : Vous planifiez et exécutez un gros refactoring

Étape 2 : Vous lancez les tests en arrière-plan

Étape 3 :

/loop 3m Vérifie les résultats des tests. Si quelque chose
échoue à cause du refactoring, corrige-le.

Claude surveille et corrige automatiquement les régressions.

Scénario 3 : Boucle de travail autonome

/loop
Il y a 12 issues ouvertes dans le repo. Pour chaque issue :
1. Planifie la correction
2. Implémente la correction
3. Fais un commit
4. Passe à l'issue suivante
Arrête-toi quand toutes les issues sont traitées.

Claude travaille de manière totalement autonome, en planifiant chaque correction avant de la coder.

Conseil final : Commencez par utiliser les plans seuls pour les tâches complexes. Quand vous êtes à l'aise, ajoutez les boucles pour automatiser les tâches répétitives. La combinaison des deux est le niveau supérieur : Claude qui réfléchit, exécute et surveille, le tout en autonomie.

Glossaire

Mode Plan
Mode de Claude Code où il analyse et propose une stratégie avant de modifier le code. Les outils d'écriture sont désactivés pendant la planification.
Boucle (/loop)
Commande qui fait répéter une tâche à Claude automatiquement, soit à intervalle fixe, soit en mode auto-régulé.
Cache de contexte
Mémoire temporaire (5 minutes) qui permet à Claude de reprendre une conversation rapidement. Influence le choix de l'intervalle des boucles.
Boucle à intervalle fixe
Boucle où vous spécifiez l'intervalle (ex : /loop 5m). Claude se réveille exactement à cet intervalle.
Boucle dynamique
Boucle où Claude choisit lui-même l'intervalle en fonction de la tâche. Lancée avec /loop sans intervalle.
Sous-agent
Instance spécialisée de Claude qui travaille sur une sous-tâche dans un contexte séparé, puis renvoie le résultat à Claude principal.
Réveil (wake-up)
Mécanisme interne qui « réveille » Claude à l'heure programmée pour la prochaine itération d'une boucle.
Espace de réflexion (thinking)
Zone interne où Claude organise ses pensées avant de répondre. Invisible pour l'utilisateur, mais essentiel pour la qualité des plans.
